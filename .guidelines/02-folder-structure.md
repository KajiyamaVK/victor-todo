# Folder Structure

## Top-Level Layout

```
taskem/
  lib/                          # All application code
    domain/                     # Layer 1: Pure Dart business logic
    data/                       # Layer 2: SQLite/drift persistence
    presentation/               # Layer 3: Flutter UI + Riverpod
    services/                   # External service clients (LiteLLM stub)
    core/                       # Shared constants and utilities
    main.dart                   # App entry point
  test/                         # All tests
    unit/
    widget/
    integration/                # Future use
  .guidelines/                  # This folder
  CLAUDE.md
  AGENTS.md
  pubspec.yaml
  analysis_options.yaml
  .env.example
  .gitignore
```

## Data Layer (`lib/data/`)

**Rule: Depends on domain (implements repository interfaces). No Flutter imports.**

```
lib/data/

    database/
      app_database.dart         # drift AppDatabase class, table definitions
      app_database.g.dart       # Generated by build_runner (do not edit)

      task_dao.dart             # drift DAO for tasks
      category_dao.dart         # drift DAO for categories
      tag_dao.dart              # drift DAO for tags
      task_list_dao.dart        # drift DAO for task lists
  mappers/

    category_mapper.dart
    tag_mapper.dart
    task_list_mapper.dart
  repositories/

    category_repository_impl.dart
    tag_repository_impl.dart
    task_list_repository_impl.dart
```

## Rules

1. Generated files (`*.g.dart`) are committed. They are only regenerated when the drift schema changes.
2. Do not create a `lib/features/` folder — this project uses layer-based (not feature-based) organization.
3. Do not add platform-specific code (e.g., `lib/platform/linux/`) — all platform differences are handled through packages.
4. `lib/main.dart` wires together the `ProviderScope`, `AppDatabase`, and `MaterialApp`. It should be minimal.
